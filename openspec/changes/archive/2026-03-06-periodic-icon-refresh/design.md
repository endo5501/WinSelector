## Context

WinSelectorは2秒ごとにウィンドウ一覧をリフレッシュしているが、アイコンは`Win32Utils::getWindowIcon()`内の静的キャッシュ(`s_iconCache`)により初回取得時の結果が永続的に保持される。これによりアイコンの変化（通知バッジ、アプリ状態変化）や初回取得失敗時のリカバリができない。

現在のアイコン取得フロー:
1. `WindowScanner::getWindows()` → `Win32Utils::getWindowIcon(hwnd)`
2. `getWindowIcon()` はキャッシュにヒットすればキャッシュから返す
3. キャッシュミスの場合のみWin32 APIでアイコンを取得しキャッシュに保存
4. キャッシュのクリアは `clearIconCache()` でのみ行われ、現在はウィンドウが閉じられた時のみ呼ばれる

## Goals / Non-Goals

**Goals:**
- 約60秒ごとにアイコンキャッシュを無効化し、アイコンの再取得を可能にする
- 既存のリフレッシュサイクル（2秒）に影響を与えない
- 設定値として更新間隔を管理する

**Non-Goals:**
- 個別ウィンドウ単位でのアイコン変更検知（差分更新）
- アイコン取得のバックグラウンドスレッド化
- アイコンの変更通知をWin32イベントで受け取る仕組み

## Decisions

### 1. MainWindowにアイコンキャッシュ無効化タイマーを追加

**選択**: MainWindowで新しいQTimerを作成し、一定間隔で`Win32Utils::clearIconCache(nullptr)`を呼び出す。

**理由**: 既存の`m_refreshTimer`とは独立したタイマーにすることで、ウィンドウリフレッシュ間隔とアイコン更新間隔を独立して制御できる。キャッシュクリア後、次の通常リフレッシュサイクル（最大2秒以内）でアイコンが再取得される。

**代替案**:
- リフレッシュ回数をカウントしてN回ごとにクリア → タイマーの方がシンプルで直感的
- Win32のアイコン変更イベント監視 → 複雑すぎ、Non-Goal

### 2. 更新間隔をconfigに追加

**選択**: `WinSelectorConfig`に`iconRefreshIntervalMs()`を追加（デフォルト60000ms）。

**理由**: 他の設定値と一貫した管理方法。将来的にユーザー設定可能にする余地を残す。

## Risks / Trade-offs

- **[全キャッシュクリアによる一時的なパフォーマンス影響]** → 60秒に1回であり、アイコン取得自体は軽量なWin32 API呼び出しなので影響は軽微。SendMessageTimeoutで200msタイムアウトも設定済み。
- **[アイコンの一瞬の点滅]** → キャッシュクリア後、次のリフレッシュで再取得されるまでの間は既にsetInfoで描画済みのアイコンが表示されるため、視覚的な問題はない。
